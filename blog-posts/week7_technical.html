T7: Injecting knowledge!

09/18/2014

Since SQL hasn’t been mentioned here yet, allow me to give you just a brief overview. Pronounced es-queue-el or ‘Sequel’ (depending on who you talk to), it stands for Structured Query Language. Essentially, it’s a language that allows you to interact and manipulate data in databases. If you’ve ever used a web app or done anything that involves a login, you’ve most likely used SQL. It’s everywhere!

And since it’s everywhere and contains loads of information, naturally there are people who will try to gain access to that information. The process that they use is called SQL injection. What happens is that SQL commands will be entered into a field(such as a login or a form) and will run as part of the database. This could potentially allow someone to bypass a login, gain access to credit card information, or delete important databases.

The Danger of 1=1

Let’s say that on a web application there is a user input for their name. If the application hasn’t been properly sanitized, someone could put their UserId as 101 OR 1=1. When this get’s run in the SQL, it’ll be run like below: 

SELECT * FROM users WHERE UserId = 101 or 1=1
Because WHERE 1=1 will always be true (because one does equals one), this would return all the rows from the table. Similarly, using “” or “=” could do the same, since “=” will return true.

Another terrifying thing that could happen would be if someone hacked in and deleted a table, and it’s surprisingly easy to do. Like in our example above, someone could tack in 101; DROP TABLE ImportantInfo; and this would create the following command in SQL: 

SELECT * FROM users WHERE userId = 101; DROP TABLE ImportantInfo;
SQL would read the first command as anything up to the semicolon, which in this case would be to return the information for userId 101. Then it would read the command after the semicolon, which is to delete table ImportantInfo.

Prevention

As a newbie, I can recognize the brilliance of injection, but am not well versed in the ways of preventing it. It seems like the best way to prevent injection would be to build parameters for SQL. Parameterized Queries essentially check to see if the users input literally matched the information. So in our above example, it would check to see if 101 or 1=1 was a user’s id.

Another way to prevent injection would be to use stored procedures, which acts similarly to prepared statements, with the difference being that stored procedures are defined and stored in the database. The procedures would then be called to the application when needed.

The third main defense against injection would be escaping all user supplied input. To do this, you would escape the users input before putting it in the query, which prevents SQL from confusing the users input with the code written by the developer. However, this method should be used with caution, as it is not as sturdy as the previous two methods.

It is essential to use preventative methods to avoid inject if you plan on using SQL. Security breaches are becoming more and more often (I’m looking at you Target), and they never end well. Knowing how to prevent it from happening in the first place is crucial, and I hope to gain more with this as I continue learning about SQL.
